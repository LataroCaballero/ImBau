# Quick Task 260626-f90: Fix red CI — pass test DB/Redis env through Turbo strict mode + add Redis service

<objective>
Make the `quality` CI check (PR #1, branch `fase-0/foundation`) pass. The only real failure is `@imbau/db#test` throwing "Missing test DB connection string: set one of TEST_DATABASE_URL / DATABASE_URL". Turbo 2.x runs in strict env mode (no `envMode`/`globalPassThroughEnv` in turbo.json), so it strips every env var not declared on the task before running `vitest`. `.github/workflows/ci.yml` exports the `TEST_DATABASE_*` URLs correctly, but Turbo strips them from the `test` task → vitest globalSetup sees them unset → throws. The repo-wide "lint failed" symptom was Turbo SIGTERM-killing in-flight tasks when `db#test` failed first; lint actually passes (verified locally 8/8 cold). Secondary: `apps/worker/src/index.test.ts` is a live-Redis BullMQ smoke test, and ci.yml has no Redis service.
</objective>

<tasks>

<task id="1">
  <name>Declare test env passthrough on turbo.json `test` task</name>
  <files>turbo.json</files>
  <action>On the `test` task object, add a `passThroughEnv` array (keep the existing `dependsOn: ["^build"]`): ["TEST_DATABASE_URL","TEST_DATABASE_APP_URL","TEST_DATABASE_ANON_URL","DATABASE_URL","DATABASE_APP_URL","DATABASE_ANON_URL","REDIS_URL","NODE_ENV"]. These are runtime connection strings, so passThroughEnv (not env) is correct — they must reach the task without becoming part of the cache key.</action>
  <verify>node -e 'const t=require("./turbo.json");const e=t.tasks.test.passThroughEnv||[];const need=["TEST_DATABASE_URL","TEST_DATABASE_APP_URL","TEST_DATABASE_ANON_URL","REDIS_URL"];if(!need.every(x=>e.includes(x)))throw new Error("missing: "+need.filter(x=>!e.includes(x)));console.log("ok")' && node -e 'JSON.parse(require("fs").readFileSync("turbo.json","utf8"))'</verify>
  <done>turbo.json `test` task declares passThroughEnv including the three TEST_DATABASE_* vars + REDIS_URL; file is valid JSON.</done>
</task>
