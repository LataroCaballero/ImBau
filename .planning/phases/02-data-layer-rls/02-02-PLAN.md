<objective>
Define the phase-2 schema and its RLS as code, then materialize it as a versioned Drizzle migration plus the hand-written role/GRANT/FORCE SQL that Drizzle cannot emit. Better Auth's `organization` (extended with `plan`) is the canonical tenant (D-01); `member` is canonical membership (D-02); `projects` carries the `estado` enum and both the tenant policy and the anon published-only policy (D-11). Roles `app_authenticated`/`anon` and `FORCE ROW LEVEL SECURITY` on every tenant table land as raw SQL in the same migration journal (FLAG-2), keeping ONE migration history (DATA-02).

Purpose: DATA-02 (versioned base schema, `generate` + `migrate`, never `push`) and the database side of DATA-03 (FORCE RLS, dedicated roles, anon published filter). This plan produces the DDL and policies; the runtime helpers and the proving tests are plan 03.
Output: `auth.ts` (CLI generator config), folded `src/schema/auth-schema.ts`, `src/schema/projects.ts`, `src/schema/roles.ts`, `src/schema/index.ts`, a generated `migrations/0000_init.sql`, a hand-written `migrations/0001_rls.sql`, and the migration journal — all applied successfully against the Compose Postgres from plan 01.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owner/migration role → tenant tables | The privileged owner runs DDL and owns the tables; without FORCE RLS it would bypass policies (owner-bypass footgun). |
| app_authenticated / anon → tenant tables | Runtime roles; RLS policies are the only thing standing between them and other tenants' rows / draft projects. |
| tenant GUC `app.current_organization_id` → policy filter | The single value that decides which tenant's rows are visible; must come from a trusted parameterized path (set in plan 03). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Information Disclosure / Elevation | owner bypasses RLS (silent leak) | mitigate | `ALTER TABLE ... FORCE ROW LEVEL SECURITY` on every tenant table (0001_rls.sql); verified via `pg_class.relforcerowsecurity = true`. |
| T-02-05 | Elevation of Privilege | runtime role too privileged | mitigate | `CREATE ROLE ... NOSUPERUSER NOBYPASSRLS NOCREATEDB NOCREATEROLE`; app role never owns tables (D-04); verified via `pg_roles.rolsuper=false AND rolbypassrls=false`. |
| T-02-06 | Information Disclosure | draft (`borrador`) project leaks to anon | mitigate | `projects_anon_published` policy `for: select` `using estado='publicado'` + anon GRANTed SELECT only; proven in plan 03 case (d). |
| T-02-07 | Information Disclosure | tenant filter never matches / type mismatch | mitigate | Policy cast (`::text`/`::uuid`) reconciled against the actual generated `organization.id` type (A1/Pitfall 2); FK typed to match. |
| T-02-08 | Tampering | second/competing migration history | mitigate | BA CLI `generate` only; CLI `migrate` (Kysely-only) never run; all DDL through drizzle-kit one journal; no `push`/manual edits. |
| T-02-09 | Information Disclosure | role password committed to repo | mitigate | No literal `PASSWORD '...'` in committed SQL (A5); injected at apply time; SOPS encryption is phase 4. |
</threat_model>
