<objective>
Define the phase-2 schema and its RLS as code, then materialize it as a versioned Drizzle migration plus the hand-written role/GRANT/FORCE SQL that Drizzle cannot emit. Better Auth's `organization` (extended with `plan`) is the canonical tenant (D-01); `member` is canonical membership (D-02); `projects` carries the `estado` enum and both the tenant policy and the anon published-only policy (D-11). The two tenant-scoped tables this phase — `projects` AND `member` (both carry `organization_id`, D-02/D-10) — get a tenant policy + `.enableRLS()` + `FORCE ROW LEVEL SECURITY`. Roles `app_authenticated`/`anon` and `FORCE ROW LEVEL SECURITY` on every tenant table land as raw SQL in the same migration journal (FLAG-2), keeping ONE migration history (DATA-02).

Purpose: DATA-02 (versioned base schema, `generate` + `migrate`, never `push`) and the database side of DATA-03/DATA-04 (FORCE RLS on every tenant table, dedicated roles, anon published filter). This plan produces the DDL and policies; the runtime helpers and the proving tests are plan 03.
Output: `auth.ts` (CLI generator config), folded `src/schema/auth-schema.ts`, `src/schema/projects.ts`, `src/schema/member-rls.ts`, `src/schema/roles.ts`, `src/schema/index.ts`, a generated `migrations/0000_init.sql`, a hand-written `migrations/0001_rls.sql` with its matching `meta/0001_snapshot.json`, and the migration journal — all applied successfully against the Compose Postgres from plan 01.
</objective>
