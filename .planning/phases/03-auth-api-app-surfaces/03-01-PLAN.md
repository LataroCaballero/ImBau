<objective>
Stand up the Phase 3 auth foundation: the Better Auth runtime in `packages/api`, the access-control roles (owner/developer/viewer), the validated auth env preset, and the `packages/api` test harness (Vitest config + Postgres-backed fixtures) that every later plan's integration tests depend on.

This plan locks the highest-risk decision of the phase (A1): the Better Auth Drizzle adapter connects via the **elevated owner pool** (`createOwnerDb`) so it can write the RLS-FORCED `organization`/`member` tables, while the application data path stays on the unprivileged `withTenant`/`withAnon` pools. The acceptance criteria force the verifier to prove org-create + invite-accept actually write rows.

Purpose: nothing in the phase (tRPC context, panel, web reads) can be built or tested until the auth runtime boots against the phase-2 schema and the test fixtures can mint a user+org+active session.
Output: `authEnv` preset, `packages/api` auth deps, `auth/{env,access-control,runtime}.ts`, `packages/api/vitest.config.ts` + `tests/{db,fixtures}.ts`, and an auth-runtime integration test proving A1.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| auth adapter → Postgres | The Better Auth adapter holds an ELEVATED (owner) connection. The threat is this pool leaking into the application data path. |
| env → runtime | Secrets (BETTER_AUTH_SECRET, RESEND_API_KEY) cross from the environment into the runtime. |
| npm registry → repo | New packages (incl. a deprecation-flagged one) are installed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | Better Auth adapter owner pool (A1) | mitigate | Owner pool is constructed ONLY inside `runtime.ts`; acceptance criteria assert `runtime.ts` never imports `appDb`/`withTenant`. ALL app reads/writes stay on the unprivileged pools (enforced in Plan 03-02). |
| T-03-02 | Elevation of Privilege | owner pool used as RLS bypass for tenant data | accept→mitigate | The owner pool writes ONLY auth/org bookkeeping tables; it is never handed to a tRPC router. Plan 03-02 routes all tenant data through `withTenant`/`withAnon`. |
| T-03-03 | Information Disclosure | env validation error leaking secret values | mitigate | Do not override `onValidationError`; the default t3-env formatter prints the variable NAME + reason, never the offending value (V7). |
| T-03-04 | Information Disclosure | secrets committed to repo | mitigate | `RESEND_API_KEY`/`BETTER_AUTH_SECRET` come from validated env only; no real secrets in the repo. SOPS encryption is Phase 4 — do not add here, but never commit a real key. |
| T-03-SC | Tampering | npm install of `@react-email/components` (deprecation-flagged) + other Phase-3 packages | mitigate | Blocking human-verify checkpoint (Task 1) confirms the deprecation-flagged package is healthy at install; no package has a postinstall script (audit confirmed). |
</threat_model>
