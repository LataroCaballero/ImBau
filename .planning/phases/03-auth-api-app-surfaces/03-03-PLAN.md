<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser â†’ panel auth handler | Credentials + session cookies cross here. |
| invitee â†’ accept-invitation | An invitation id from an email link is consumed here. |
| panel RSC â†’ tRPC server caller | The dashboard read crosses into the tenant-scoped data path. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Spoofing | session cookie not set / not persisting | mitigate | `nextCookies()` is the last plugin (Plan 03-01); the e2e asserts the session persists across a reload (AUTH-01). |
| T-03-11 | Information Disclosure | session cookie not httpOnly/secure | mitigate | Better Auth default cookie flags (httpOnly; secure in prod); no custom cookie handling. |
| T-03-12 | Elevation of Privilege | auth handler mounted in apps/web (anon surface) | mitigate | Handler mounted ONLY in panel (D-03); acceptance criteria assert no auth route under apps/web. |
| T-03-13 | Spoofing | invitation link guessing | mitigate | Org-plugin invitation ids are non-sequential; accept requires an authenticated session (signup/login first â€” D-10). |
| T-03-14 | Information Disclosure | real invitation email/secret in dev | accept | Dev logs only the accept link (no secret); real Resend delivery is staging-only with a validated key (D-09). |
| T-03-15 | Elevation of Privilege | non-owner reaching the invite mutation | mitigate | `member.invite` is `requireRole("owner")` (Plan 03-02); the invite form is owner-gated server-side. |
</threat_model>
