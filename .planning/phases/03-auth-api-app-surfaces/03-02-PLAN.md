<objective>
Build the tRPC v11 API layer in `packages/api`: the typed init (public/protected procedures), the context that reads the Better Auth session and derives the tenant from `session.activeOrganizationId` server-side, the `requireRole` middleware, the minimal D-07 router set (`projects.listForOrg`/`listPublished`, `org.list`/`setActive`, `member.invite`, `invitation.accept`), the real `@imbau/api` barrel, and the integration tests that prove tenant isolation and role enforcement through the tRPC caller.

This plan implements the load-bearing seam: `session.activeOrganizationId → withTenant(orgId)` for protected reads/writes and `withAnon()` for the anonymous published-only path. The tenant is ALWAYS derived server-side; a client-supplied orgId is never trusted.

Purpose: the panel dashboard (APP-01) and web listing (APP-02) read through this layer; AUTH-02 (active-org tenant scoping + roles) and AUTH-03 (owner-only invite + accept) are enforced here.
Output: `trpc/{init,context,middleware}.ts`, four routers + `_app.ts`, the real `index.ts` barrel, and `tests/trpc-tenant.test.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → tRPC | Untrusted procedure input crosses here (invite email/role, setActive orgId). |
| tRPC context → DB | The tenant GUC is set from the session here; the only place orgId is authoritative. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Tampering / Information Disclosure | cross-tenant read via forged/client-supplied orgId | mitigate | `protectedProcedure` derives `activeOrgId` ONLY from `session.activeOrganizationId` server-side; routers never read an orgId from input. RLS `withTenant` enforces at the DB (defense in depth). Integration test asserts cross-tenant absence through the caller. |
| T-03-06 | Elevation of Privilege | non-owner invites members / changes roles | mitigate | `member.invite` is `requireRole("owner")`; the test proves FORBIDDEN for viewer/developer. |
| T-03-07 | Tampering | input injection at the tRPC boundary | mitigate | Zod 4 validation on every procedure input (invite email/role, setActive/accept ids). |
| T-03-08 | Spoofing / Information Disclosure | tenant GUC bleeding across pooled connections | mitigate | `withTenant` uses transaction-scoped `set_config(...,true)` (phase 2); never `SET` session-level — the GUC auto-clears at commit. |
| T-03-09 | Elevation of Privilege | a router importing the owner pool / appDb directly | mitigate | Routers import ONLY `withTenant`/`withAnon`/`schema` from `@imbau/db`; grep confirms no `createOwnerDb`/`appDb` import in any router. |
</threat_model>
